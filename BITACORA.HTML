PROYECTO FINAL DE FUNDAMENTOS DE SOFTWARE

1. Clone (clonar repositorio)

Acción de descargar una copia completa de un repositorio Git en tu equipo para trabajar en él.

2. Pull (traer cambios)

Acción en Git para traer y combinar actualizaciones desde un repositorio remoto hacia el local.

3. Push (enviar cambios)

Acción en Git para enviar tus commits locales al repositorio remoto.

4. Merge (fusión)

Unir los cambios de una rama a otra en Git, integrando diferentes líneas de desarrollo.

5. Rebase

Reaplicar commits de una rama sobre otra para mantener un historial más lineal.

6. Conflict (conflicto en Git)

Ocurre cuando dos cambios se hacen en la misma parte del código y Git no puede decidir cuál mantener.

7. Repositorio remoto

Espacio en la nube (ej. GitHub, GitLab) donde se almacenan los proyectos para compartir y colaborar.

8. Repositorio local

Copia de un repositorio que está en tu computadora, donde puedes trabajar antes de subir cambios.

9. Tag (etiqueta en Git)

Marca especial que señala un punto importante en la historia del proyecto, como una versión estable.

10. Release (versión publicada)

Conjunto de cambios empaquetados como una versión oficial para que los usuarios puedan instalarla.

11. .gitignore

Archivo que indica a Git qué elementos (ej. contraseñas, logs, temporales) no deben incluirse en el repositorio.

12. Continuous Integration (CI)

Práctica de integrar código frecuentemente en un repositorio común, verificando automáticamente que todo funcione.

13. Continuous Delivery (CD)

Proceso que permite desplegar software de manera automática y frecuente en entornos de pruebas o producción.

14. Pipeline (tubería de CI/CD)

Serie de pasos automáticos (compilar, probar, desplegar) que aseguran calidad en el software.

15. Webhook

Método para recibir notificaciones automáticas cuando ocurre un evento en un servicio (ej. push en GitHub).

16. Pull Request (PR)

Solicitud para que los cambios de una rama sean revisados y fusionados en la rama principal.

17. Code Review (revisión de código)

Proceso en el que compañeros revisan cambios antes de ser aceptados, para detectar errores y mejorar calidad.

18. Linting

Análisis automático del código para encontrar errores de estilo, sintaxis o prácticas inseguras.

19. Static Analysis (análisis estático)

Evaluación del código sin ejecutarlo, detectando vulnerabilidades o malas prácticas.

20. Dynamic Analysis (análisis dinámico)

Evaluación del software en ejecución para encontrar errores o problemas de rendimiento.

21. Unit Test (prueba unitaria)

Test que verifica el funcionamiento correcto de una parte pequeña del código, como una función.

22. Integration Test (prueba de integración)

Test que comprueba si varios módulos funcionan correctamente al combinarse.

23. End

to-End Test (prueba extremo a extremo): Verifica que un sistema completo funcione simulando la experiencia real del usuario.

24. Test Coverage (cobertura de pruebas)

Métrica que mide qué porcentaje del código está cubierto por pruebas automáticas.

25. Mock (simulación)

Objeto o función falsa usada en pruebas para simular comportamientos sin depender de sistemas reales.

26. CI/CD runner

Servidor que ejecuta automáticamente las pruebas y despliegues configurados en un pipeline.

27. Issue (incidencia)

Registro de un problema, tarea o mejora en un proyecto de software.

28. Kanban board

Tablero visual usado en metodologías ágiles para organizar tareas en columnas como “Pendiente”, “En progreso” y “Hecho”.

29. Sprint (en Scrum)

Periodo corto de trabajo (generalmente 2,4 semanas) en el que se desarrollan tareas específicas.

30. Scrum Master

Rol en Scrum que guía al equipo en el cumplimiento de la metodología y elimina obstáculos.

31. Product Owner

Rol encargado de definir prioridades del producto y gestionar el backlog en metodologías ágiles.

32. Backlog

Lista priorizada de tareas, funciones o mejoras pendientes en un proyecto.

33. Daily stand

Reunión corta diaria donde el equipo comenta avances, bloqueos y próximos pasos.

34. Retrospectiva

Reunión al final de un sprint para evaluar qué salió bien, qué mejorar y cómo aplicar cambios.

35. Burndown chart

Gráfico que muestra el progreso de un sprint y cuánto trabajo queda por completar.

36. Velocity (en Scrum)

Medida de la cantidad de trabajo que un equipo puede completar en un sprint, útil para planificar.

37. Metodología ágil

Conjunto de prácticas que promueven entregas rápidas, comunicación constante y adaptación a cambios.

38. Modelo en cascada

Método clásico de desarrollo donde se completan fases secuenciales (análisis, diseño, implementación, pruebas).

39. Modelo incremental

Estrategia donde el software se construye en partes, entregando versiones funcionales poco a poco.

40. Modelo iterativo

Método en el que se repite el ciclo de desarrollo varias veces hasta perfeccionar el sistema.

41. MVP (Producto Mínimo Viable)

Versión básica de un producto que incluye lo necesario para validar una idea con usuarios reales.

42. Prototipo

Versión preliminar de un sistema que permite evaluar ideas y detectar mejoras antes de la implementación final.

43. Viabilidad

Estudio para determinar si un proyecto es factible en términos técnicos, económicos y de tiempo.

44. Alcance (scope)

Definición de qué incluye y qué no incluye un proyecto, clave para evitar desviaciones.

45. Deadline

Fecha límite establecida para entregar una tarea o proyecto.

46. Kick-off

Reunión inicial donde se define la visión, roles y metas de un proyecto.

47. Stakeholder (parte interesada)

Persona o entidad que tiene interés en el éxito de un proyecto, como usuarios, clientes o gerentes.

48. Gantt chart

Herramienta visual para planificar proyectos mostrando tareas y su duración en un calendario.

49. Ruta crítica

Secuencia de tareas que determinan la duración mínima de un proyecto.

50. Gestión de riesgos

Identificación y mitigación de posibles problemas que puedan afectar un proyecto.

51. Plan de contingencia

Estrategia preparada para responder a imprevistos y minimizar impacto en el proyecto.

52. Planificación iterativa

Ajuste continuo del plan de proyecto según nuevas condiciones o aprendizajes.

53. Indicadores KPI

Métricas clave usadas para medir el rendimiento de un proyecto o sistema.

54. Balanced Scorecard (Cuadro de Mando Integral)

Método de gestión que mide rendimiento en distintas perspectivas (financiera, clientes, procesos, aprendizaje).

55. OKR (Objectives and Key Results)

Sistema de metas medibles que ayudan a alinear equipos y organizaciones.

56. Benchmarking

Comparación de procesos o sistemas con los mejores del sector para encontrar oportunidades de mejora.

57. Monitoreo de sistemas

Supervisión continua de servidores, redes y aplicaciones para detectar problemas.

58. Logging (registro)

Almacenamiento de eventos y errores que ocurren en un sistema para analizarlos después.

59. Alertas

Notificaciones automáticas que informan cuando ocurre un problema en un sistema.

60. Observabilidad

Capacidad de un sistema para mostrar su estado interno a través de métricas, logs y trazas.

61. Trazas (tracing)

Seguimiento de cómo viaja una petición a través de múltiples servicios en un sistema distribuido.

62. Dashboard

Panel visual que muestra métricas y datos en tiempo real para facilitar decisiones rápidas.

63. SLA (Service Level Agreement)

Acuerdo entre proveedor y cliente que define niveles de servicio esperados (ej. disponibilidad).

64. SLO (Service Level Objective)

Objetivo específico de nivel de servicio que una empresa se compromete a cumplir.

65. SLI (Service Level Indicator)

Métrica usada para medir un aspecto del servicio, como latencia o disponibilidad.

66. MTTR (Mean Time To Repair)

Tiempo promedio que tarda un sistema en recuperarse tras un fallo.

67. MTBF (Mean Time Between Failures)

Tiempo promedio que pasa entre fallos de un sistema o dispositivo, usado para medir su confiabilidad.

68. Alta disponibilidad

Configuración de sistemas diseñada para que sigan funcionando incluso si falla un componente.

69. Escalabilidad vertical

Incrementar la capacidad de un sistema añadiendo más recursos a un solo servidor (ej. más RAM o CPU).

70. Escalabilidad horizontal

Incrementar capacidad añadiendo más servidores que trabajan juntos, común en sistemas en la nube.

71. Balanceo de carga

Distribución del tráfico entre varios servidores para evitar sobrecarga y mejorar rendimiento.

72. Cluster de servidores

Conjunto de servidores que funcionan como una sola unidad para ofrecer más potencia y tolerancia a fallos.

73. Contenedor

Entorno aislado que ejecuta aplicaciones con sus dependencias sin afectar al resto del sistema.

74. Docker

Plataforma que permite crear y gestionar contenedores de software de manera eficiente.

75. Imagen de contenedor

Plantilla que contiene todo lo necesario para ejecutar una aplicación en un contenedor.

76. Orquestación de contenedores

Técnica que gestiona automáticamente múltiples contenedores, asegurando que estén disponibles.

77. Kubernetes

Herramienta de orquestación de contenedores muy usada para administrar aplicaciones en la nube.

78. Microservicio

Arquitectura en la que una aplicación se divide en pequeños servicios independientes que se comunican entre sí.

79. Monolito

Aplicación construida como un solo bloque de software, más difícil de escalar que los microservicios.

80. API Gateway

Punto de entrada centralizado que gestiona el tráfico entre clientes y microservicios.

81. Service Mesh

Capa de software que controla cómo interactúan los microservicios entre sí.

82. Escalado automático

Ajuste dinámico de recursos (más servidores o menos) según la demanda de usuarios.

83. Cloud híbrida

Combinación de nube pública y privada, permitiendo flexibilidad y control.

84. Cloud pública

Servicios de nube accesibles a cualquiera mediante suscripción (ej. AWS, Azure, Google Cloud).

85. Cloud privada

Infraestructura de nube exclusiva de una organización, más segura y controlada.

86. Edge computing

Procesamiento de datos cerca del lugar donde se generan, reduciendo latencia.

87. Fog computing

Extensión de la computación en la nube que procesa datos en dispositivos intermedios, no solo en servidores centrales.

88. Servidor físico

Computadora dedicada que proporciona servicios de hardware y software a otros dispositivos.

89. Servidor virtual

Máquina creada mediante software que simula ser un servidor real.

90. Hipervisor

Software que permite crear y gestionar máquinas virtuales en un servidor físico.

91. VM (Máquina Virtual)

Entorno virtual que emula un sistema operativo completo dentro de otro.

92. Snapshot (instantánea)

Copia rápida del estado de una máquina virtual para restaurarla después si ocurre un error.

93. Backup en la nube

Copias de seguridad almacenadas en servidores remotos accesibles por Internet.

94. Respaldo incremental

Copia que guarda solo los cambios hechos desde la última copia, optimizando espacio.

95. Respaldo diferencial

Copia que guarda los cambios desde la última copia completa, más rápida de restaurar.

96. Disaster Recovery (recuperación de desastres)

Estrategias y procesos para recuperar sistemas tras fallos graves.

97. Plan de continuidad

Conjunto de medidas para mantener un sistema operativo aunque ocurran incidentes.

98. Failover

Cambio automático a un sistema de respaldo cuando falla el principal.

99. Redundancia

Duplicación de componentes críticos para garantizar disponibilidad continua.

100. Hot site

Centro de respaldo que está listo para operar de inmediato en caso de fallo del principal.

101. Cold site

Centro de respaldo que requiere instalación y configuración antes de usarse.

102. DNS (Domain Name System)

Sistema que traduce nombres de dominio (ej. google.com) en direcciones IP.

103. Proceso De Elt

Concepto de ingeniería de software orientado a mejorar calidad, mantenibilidad y entrega de valor.

104. Cache DNS

Memoria temporal que guarda resoluciones recientes de nombres de dominio para acelerar accesos.

105. TTL (Time To Live en DNS)

Tiempo que un registro DNS permanece en caché antes de actualizarse.

106. Dominio

Nombre único que identifica un sitio web en Internet.

107. Subdominio

Parte de un dominio que organiza o divide secciones de un sitio (ej. blog.ejemplo.com).

108. Hosting

Servicio que almacena los archivos de un sitio web en servidores conectados a Internet.

109. Servidor web

Software que entrega páginas web a los navegadores (ej. Apache, Nginx).

110. Navegador

Programa que permite visualizar páginas web (ej. Chrome, Firefox, Edge).

111. Motor de búsqueda

Sistema en línea que ayuda a encontrar información en Internet (ej. Google).

112. SEO (Search Engine Optimization)

Prácticas para mejorar la visibilidad de un sitio web en buscadores.

113. SEM (Search Engine Marketing)

Estrategias de publicidad pagada para destacar en motores de búsqueda.

114. Landing Page

Página diseñada con un objetivo específico, como ventas o registro de usuarios.

115. Responsive Design

Diseño web que se adapta automáticamente a distintos tamaños de pantalla.

116. Accesibilidad web

Conjunto de prácticas para que los sitios sean usables por personas con discapacidades.

117. HTML5

Versión moderna del lenguaje HTML que incorpora audio, video y nuevos elementos semánticos.

118. CSS3

Última versión de CSS, que añade animaciones, transiciones y estilos avanzados.

119. Flexbox

Módulo de CSS que permite organizar elementos en contenedores flexibles.

120. Grid Layout

Sistema de CSS que organiza elementos en filas y columnas, muy usado en interfaces modernas.

121. Media Query

Regla de CSS que adapta estilos según el tamaño de la pantalla.

122. JavaScript moderno (ES6+)

Conjunto de mejoras del lenguaje JavaScript, como arrow functions y módulos.

123. DOM (Document Object Model)

Representación estructurada de una página web que JavaScript puede modificar.

124. Evento en JavaScript

Acción que ocurre en una página (ej. clic del usuario) y puede activar funciones.

125. Ajax

Técnica que permite actualizar contenido en una página sin recargarla completa.

126. Fetch API

Método moderno de JavaScript para hacer solicitudes HTTP y obtener datos de un servidor.

127. JSON (JavaScript Object Notation)

Formato de datos ligero y fácil de leer, usado para intercambiar información entre sistemas.

128. XML (eXtensible Markup Language)

Lenguaje usado para representar datos de forma estructurada.

129. YAML

Formato de datos simple, usado en configuración de sistemas y aplicaciones.

130. Markdown

Lenguaje ligero para dar formato a texto, muy usado en documentación y GitHub.

131. WYSIWYG

Acrónimo de “What You See Is What You Get”, editores donde lo que ves en pantalla es lo que obtienes al imprimir o guardar.

132. CMS (Content Management System)

Software que permite crear y gestionar sitios web fácilmente (ej. WordPress).

133. WordPress

CMS muy popular que permite crear blogs y sitios web sin necesidad de programar.

134. Joomla

Otro CMS de código abierto usado para crear sitios web y aplicaciones en línea. 140. Drupal: CMS flexible usado para sitios web complejos y corporativos. 141. Plantilla web: Diseño predefinido que se aplica a un sitio web para personalizar su apariencia.

135. Plugin

Complemento que añade nuevas funciones a un programa o CMS.

136. Widget

Elemento visual pequeño que muestra información o funciones en un sitio web o aplicación.

137. Formulario web

Espacio en una página donde el usuario ingresa datos que se envían a un servidor.

138. Captcha

Prueba que distingue a usuarios humanos de bots en formularios web.

139. SSL (Secure Sockets Layer)

Protocolo que cifra la comunicación entre navegador y servidor web.

140. Certificado SSL

Archivo digital que valida la seguridad de un sitio web y activa HTTPS.

141. HTTPS

Versión segura de HTTP que protege la comunicación contra espionaje y robo de datos.

142. Ciberseguridad

Conjunto de prácticas y tecnologías para proteger sistemas informáticos y datos.

143. Vulnerabilidad

Debilidad en un sistema que puede ser explotada por atacantes.

144. Exploit

Código o técnica que aprovecha una vulnerabilidad para realizar un ataque.

145. Parche de seguridad

Actualización que corrige fallos en software para evitar ataques.

146. Actualización de software

Proceso de instalar nuevas versiones con mejoras o correcciones.

147. Mantenimiento preventivo

Acciones planificadas para evitar que un sistema falle.

148. Mantenimiento correctivo

Reparación de sistemas una vez que ocurre una falla.

149. Mantenimiento predictivo

Uso de sensores y análisis de datos para anticipar cuándo fallará un sistema.

150. Diagnóstico

Identificación de problemas en un sistema informático.

151. Soporte técnico

Servicio que ayuda a los usuarios a resolver problemas con sus dispositivos o programas.

152. Mesa de ayuda (Help Desk)

Equipo o software que gestiona solicitudes de soporte técnico.

153. Ticket de soporte

Registro que documenta un problema reportado y su proceso de solución.

154. Documentación técnica

Conjunto de manuales, guías y referencias que explican el uso y mantenimiento de un sistema.

155. Manual de usuario

Documento que explica cómo usar un sistema o aplicación desde la perspectiva del usuario.

156. Manual técnico

Documento dirigido a personal especializado que explica instalación, configuración y mantenimiento de un sistema.

157. Especificación de requisitos

Documento que detalla qué debe cumplir un software antes de desarrollarlo.

158. Caso de uso

Descripción de cómo un usuario interactúa con un sistema para lograr un objetivo.

159. Diagrama de flujo

Representación gráfica de los pasos de un proceso o algoritmo.

160. Pseudocódigo

Forma intermedia entre lenguaje humano y programación para representar algoritmos.

161. Sintaxis

Conjunto de reglas que definen cómo deben escribirse las instrucciones en un lenguaje de programación.

162. Variable

Espacio en memoria donde se almacena un valor que puede cambiar durante la ejecución de un programa.

163. Constante

Valor fijo que no cambia durante la ejecución de un programa.

164. Tipo de dato

Clasificación de los valores que puede almacenar una variable, como número entero o texto.

165. Entero

Tipo de dato que representa números sin decimales.

166. Decimal

Tipo de dato que representa números con punto decimal.

167. Cadena de texto

Tipo de dato que almacena secuencias de caracteres como letras y símbolos.

168. Booleano

Tipo de dato lógico que solo puede ser verdadero o falso.

169. Arreglo (array)

Conjunto de elementos del mismo tipo almacenados en posiciones consecutivas de memoria.

170. Lista

Estructura de datos que almacena varios elementos en orden y permite manipularlos fácilmente.

171. Cola (queue)

Estructura de datos donde el primer elemento en entrar es el primero en salir (FIFO).

172. Pila (stack)

Estructura de datos donde el último en entrar es el primero en salir (LIFO).

173. Matriz

Arreglo bidimensional que organiza datos en filas y columnas.

174. Registro

Conjunto de campos relacionados que representan una entidad, como un cliente o producto.

175. Clave primaria

Identificador único de un registro en una tabla de base de datos.

176. Clave foránea

Campo que conecta una tabla con otra en una base de datos.

177. Consulta SQL

Instrucción que obtiene, modifica o elimina información en una base de datos.

178. Insert (SQL)

Comando que añade nuevos registros a una tabla.

179. Update (SQL)

Comando que modifica datos existentes en una tabla.

180. Delete (SQL)

Comando que elimina registros de una tabla.

181. Select (SQL)

Comando que recupera información de una base de datos.

182. Filtro (WHERE en SQL)

Condición que limita qué registros se muestran o afectan en una consulta.

183. Ordenamiento (ORDER BY)

Comando SQL que organiza los resultados según un criterio.

184. Agrupamiento (GROUP BY)

Comando que organiza registros similares en grupos para análisis.

185. Función de agregación

Operación en SQL que resume datos, como SUMA o PROMEDIO.

186. Índice en base de datos

Estructura que acelera la búsqueda de registros en una tabla.

187. Transacción

Conjunto de operaciones SQL que se ejecutan como una sola unidad de trabajo.

188. Rollback

Comando que revierte los cambios hechos en una transacción no confirmada.

189. Normalización

Proceso de organizar datos en una base de datos para evitar redundancia.

190. Desnormalización

Técnica de optimización que introduce redundancia para acelerar consultas.

191. Backup de base de datos

Copia de seguridad de la información contenida en una base de datos.

192. Restauración de base de datos

Proceso de recuperar la información desde una copia de seguridad.

193. Motor de base de datos

Software que gestiona el almacenamiento, consulta y seguridad de la información.

194. TICs (Tecnologías de la Información y la Comunicación)

Conjunto integrado de tecnologías —hardware, software, redes y servicios— que permiten crear, procesar, almacenar y transmitir información. Incluye herramientas de comunicación (correo, mensajería), plataformas educativas y sistemas administrativos; su comprensión es básica para diseñar soluciones prácticas en entornos educativos y empresariales.

195. Software

Conjunto de programas, librerías y datos que indican al hardware qué operaciones realizar. Va desde sistemas operativos hasta aplicaciones de usuario; la calidad del software, su modularidad y sus dependencias influyen directamente en la estabilidad, seguridad y mantenibilidad de un sistema.

196. Hardware

Elementos físicos de un sistema informático (CPU, memoria, almacenamiento, placa base, periféricos). Su arquitectura y especificaciones (velocidad, capacidad, compatibilidad) determinan qué software puede ejecutarse eficientemente y qué tipo de tareas son posibles.

197. Internet

Red global que interconecta millones de dispositivos mediante protocolos estándar (ej. TCP/IP). Permite servicios como la web, correo y streaming; su funcionamiento se basa en el enrutamiento de paquetes, resolución de nombres (DNS) y capas de protocolos que aseguran interoperabilidad.

198. Lúmenes (de un proyector)

Medida del flujo luminoso total que emite el proyector; determina la luminosidad de la imagen proyectada. Al elegir un proyector para un aula o salón, hay que considerar lúmenes junto con la resolución y la iluminación ambiental para garantizar visibilidad.

199. Overclocking

Práctica de aumentar la frecuencia de reloj de un componente (CPU/GPU) por encima de su especificación para obtener más rendimiento. Implica mayor consumo energético y calor, por lo que necesita mejor refrigeración; si no se hace con cuidado, puede causar inestabilidad o daño al hardware.

200. Renderizar

Convertir modelos, escenas o contenido en bruto a imágenes o frames finales visibles. En gráficos 3D y videojuegos puede ser en tiempo real (priorizando velocidad) o por lotes (priorizando calidad); el proceso usa recursos de CPU/GPU, memoria y VRAM.

201. VRAM (Video RAM)

Memoria dedicada en la tarjeta gráfica utilizada para almacenar texturas, imágenes intermedias (framebuffers) y datos de renderizado. Tener suficiente VRAM permite trabajar con resoluciones altas y texturas detalladas sin que el rendimiento se degrade por intercambio con memoria lenta.

202. Computadoras cuánticas

Sistemas experimentales que usan qubits y fenómenos como superposición y entrelazamiento para procesar ciertos problemas con potencial de alta ventaja respecto a la computación clásica. A día de hoy son tecnologías de investigación con desafíos como el ruido, la corrección de errores y la refrigeración.

203. Protocolo IP (Internet Protocol)

Conjunto de reglas que define cómo se estructuran, direccionan y enrutan los paquetes a través de redes. IP determina identificadores (direcciones) y fragmentación de datos; trabaja en la capa de red del modelo OSI para permitir entrega punto a punto.

204. Protocolo HTTP (HyperText Transfer Protocol)

Protocolo de aplicación que define cómo los clientes (navegadores) y servidores intercambian recursos en la web. Incluye métodos (GET, POST, PUT, DELETE) y códigos de estado (200, 404) que informan sobre el resultado de las peticiones.

205. Punto final (Endpoint) de API

URL o recurso concreto en una API que expone una operación o un dato (por ejemplo, /usuarios/123). Los endpoints definen entradas y salidas (por ejemplo, JSON), métodos HTTP permitidos y reglas de autenticación/validación.

206. Realidad virtual (VR)

Tecnología que crea ambientes digitales inmersivos mediate visores y sensores, reemplazando la percepción sensorial del usuario. Su efectividad depende de la baja latencia, alta tasa de refresco y seguimiento preciso de movimiento para evitar mareos y aumentar la sensación de presencia.

207. Realidad aumentada (AR)

Superposición de información digital sobre la vista del mundo real (texto, imágenes u objetos 3D) para enriquecer la interacción. Se aplica en educación, mantenimiento y servicios, y se ejecuta en dispositivos móviles o gafas con sensores y cámaras.

208. Arduino (plataforma)

Ecosistema de placas con microcontroladores, librerías y un IDE sencillo orientado al prototipado electrónico. Ideal para aprender programación embebida y construir proyectos con sensores y actuadores controlados por código.

Conjuntos de datos públicos y reutilizables que fomentan transparencia, innovación y creación de servicios; requieren estandarización y documentación para ser aprovechables.

345. Debugging (depuración)

Proceso sistemático de localizar y corregir errores en el código usando técnicas como logs, breakpoints y pruebas unitarias; buena disciplina reduce tiempo de resolución.

346. Deepfake (concepto y riesgos)

Contenido multimedia manipulado con IA para suplantar voces o rostros; plantea riesgos de desinformación y suplantación de identidad, por lo que se desarrollan técnicas de detección.

347. Deep Learning (aprendizaje profundo)

Subcampo de machine learning que usa redes neuronales profundas para aprender representaciones complejas a partir de datos; requiere grandes cantidades de datos y capacidad computacional.

348. Default (valor por defecto)

Valor predefinido que toma una variable o configuración si el usuario no especifica otro; elegir valores seguros por defecto mejora usabilidad y seguridad.

349. Delay (retardo)

Tiempo de espera intencional o inherente en procesos; reducir retardos en interfaces y redes mejora la experiencia del usuario.

350. Demo (demostración)

Presentación funcional de un prototipo o producto para mostrar características clave; planificar casos de uso y manejar errores esperados mejora la percepción.

351. Denegación de servicio (DoS)

Ataque que busca saturar recursos (ancho de banda, CPU) para impedir acceso legítimo a un servicio; mitigación incluye firewalls, balanceo y filtros.

352. Dependencia (software)

Librería o componente externo que un proyecto necesita; gestionarlas con control de versiones y lockfiles evita conflictos y roturas en builds.

353. Deployment (despliegue)

Proceso de pasar una aplicación desde entorno de desarrollo a producción; automatizar con pipelines CI/CD reduce errores y facilita rollbacks.

354. Desarrollador

Profesional que diseña, implementa y prueba software; puede especializarse en front-end, back-end, full-stack o áreas como DevOps y QA.

355. Desarrollo ágil

Metodología basada en iteraciones cortas, feedback frecuente y adaptabilidad; marcos como Scrum y Kanban fomentan entregas incrementales y participación del cliente.

356. Desarrollo en cascada (waterfall)

Modelo tradicional secuencial con fases definidas (requisitos, diseño, implementación, pruebas); es menos flexible ante cambios que el enfoque ágil.

357. Desinstalar

Eliminar un programa del sistema, incluyendo archivos y configuraciones residuales; limpiar restos evita conflictos si se reinstala otra versión.

358. Desktop (escritorio)

Entorno gráfico principal en sistemas operativos donde el usuario organiza ventanas, iconos y accesos directos para trabajar.

359. Device (dispositivo)

Equipo físico conectado a un sistema o red (sensores, cámaras, computadores); su gestión incluye inventario, seguridad y políticas de actualización.

360. Dial up (historia)

Tecnología antigua de conexión a Internet por línea telefónica; su estudio ayuda a entender la evolución hacia ADSL, cable y fibra.

361. Digitalización

Proceso de convertir información física en formato digital; implica captura, estructuración y aseguramiento de calidad para que los datos sean reutilizables.

362. Dimensión (en análisis de datos)

Atributo por el que se agrupan o segmentan datos (tiempo, región, producto); útil para análisis OLAP y creación de informes.

363. Proxy

Servidor intermedio que recibe peticiones de clientes y las reenvía a destinos finales; se usa para caching, anonimato, filtrado de contenido y reducción de carga en servidores origen.

364. NAT (Network Address Translation)

Técnica que permite a una red local usar direcciones IP privadas y compartir una sola IP pública al conectarse a Internet; mejora el uso de direcciones y aporta una capa básica de aislamiento.

365. Subnet (subred)

División lógica de una red mayor en segmentos más pequeños; facilita organización, mejora rendimiento y permite aplicar políticas de seguridad por segmento.

366. Máscara de subred

Número que define qué parte de una dirección IP corresponde a la red y cuál al host; esencial para determinar si dos dispositivos están en la misma subred.

367. Switch (conmutador)

Dispositivo de red que conecta múltiples equipos en una LAN y envía tramas solo al puerto destino conocido, reduciendo colisiones y mejorando eficiencia frente a un hub.

368. Router (enrutador)

Dispositivo que dirige paquetes entre redes diferentes usando tablas de ruta; conecta subredes, Internet y suele ofrecer NAT y funciones de seguridad básicas.

369. DHCP (Dynamic Host Configuration Protocol)

Servicio que asigna automáticamente direcciones IP y parámetros de red a los dispositivos, simplificando la configuración en redes con muchos equipos.

370. MAC address (dirección física)

Identificador único asignado a la interfaz de red de un dispositivo; opera a nivel de enlace de datos y se usa para comunicaciones dentro de una LAN.

371. Ping (herramienta)

Utilidad que envía paquetes ICMP para medir si un host es accesible y la latencia de ida y vuelta; básica para diagnosticar problemas de conectividad.

372. Traceroute (traza de ruta)

Herramienta que muestra el camino de los paquetes hasta un destino y los tiempos por salto, útil para localizar puntos de congestión o fallos en la ruta.

373. QoS (Quality of Service)

Conjunto de técnicas para priorizar cierto tipo de tráfico (voz, video) en redes, garantizando rendimiento en aplicaciones sensibles a latencia.

374. VLAN (Virtual LAN)

Segmento de red lógico creado dentro de un switch para separar tráfico por grupos funcionales, mejorando seguridad y organización sin necesidad de cableado físico adicional.

375. SSID (Service Set Identifier)

Nombre que identifica una red Wi-Fi; los clientes lo usan para seleccionar a qué punto de acceso conectarse.

376. WPA/WPA2/WPA3

Protocolos de seguridad para redes Wi-Fi que cifran la comunicación; WPA3 es la versión más reciente con mejoras de seguridad frente a ataques modernos.

377. SSID oculto (pros y contras)

Ocultar el SSID no es una medida de seguridad fuerte (solo oculta nombre), puede complicar la conexión de dispositivos legítimos y no evita ataques decididos.

378. NFC (Near Field Communication)

Comunicación inalámbrica de corto alcance que permite pagos y emparejamiento rápido; su seguridad depende de la capa de aplicación y del entorno del usuario.

379. RFID (Radio Frequency Identification)

Tecnología para identificar objetos por radiofrecuencia (tags y lectores); usada en inventarios, control de acceso y trazabilidad.

380. IoT (Internet of Things / Internet de las cosas)

Ecosistema de dispositivos conectados que recopilan y comparten datos; incluye sensores, actuadores, gateways y servicios en la nube para análisis y control.

381. Gateway (puerta de enlace IoT)

Dispositivo que conecta sensores y nodos IoT a la nube, traduce protocolos y realiza procesamiento local (edge) cuando es necesario.

382. MQTT (protocolo liviano)

Protocolo de mensajería ligero ideal para IoT; usa modelo publish/subscribe, consume poca ancho de banda y es eficiente para dispositivos con recursos limitados.

383. CoAP (Constrained Application Protocol)

Protocolo similar a HTTP pero diseñado para dispositivos con recursos limitados (IoT); usa UDP y simplifica operaciones RESTful en entornos embebidos.

384. On/Off Keying (OOK) — señal digital simple

Ejemplo de modulación muy básica donde la presencia o ausencia de señal representa bits; útil para comprender representación física de datos.

385. IMU (Inertial Measurement Unit)

Sensor que combina acelerómetro, giroscopio y a veces magnetómetro para medir orientación y movimiento en robótica y dispositivos móviles.

386. Acelerómetro

Sensor que mide aceleración en uno o varios ejes; se usa para detectar orientación, golpes o gestos en dispositivos portátiles.

387. Giroscopio

Sensor que mide velocidad angular y ayuda a estimar rotaciones; complementa acelerómetro para un tracking más preciso.

388. Magnetómetro

Sensor que mide campo magnético, útil para brújula electrónica y corrección de orientación en dispositivos móviles.

389. BMP / BME (sensores de presión)

Sensores que miden presión atmosférica y, en el caso de BME, también temperatura y humedad; sirven para previsión meteorológica en proyectos.

390. MQ (sensores de gas)

Familia de sensores que detectan gases como CO, LPG o metano; requieren calibración y circuitos adecuados para lecturas fiables.

391. Electret microphone

Micrófono pequeño usado con Arduino y sistemas embebidos para capturar sonido; normalmente requiere preamplificación y acondicionamiento de señal.

392. Sensor de corriente (ACS712)

Módulo para medir corriente AC/DC en un circuito; útil para monitoreo de consumo en proyectos de energía.

393. Power supply (fuente de alimentación)

Unidad que convierte la corriente de la red a voltajes y corrientes adecuados para componentes (5V/12V); su calidad afecta estabilidad del sistema.

394. Regulador de voltaje (LM7805, etc.)

Componente que entrega un voltaje estable (ej. 5V) a partir de una entrada variable; protege dispositivos sensibles frente a fluctuaciones.

395. Buck converter (conmutado)

Regulador eficiente que reduce voltaje manteniendo alta eficiencia, útil para alimentar proyectos con baterías.

396. Boost converter

Contrario del buck; eleva voltaje a partir de uno más bajo (por ejemplo usar 3.7V de batería para obtener 5V), usado en sistemas portátiles.

397. MOSFET (transistor de potencia)

Transistor usado para conmutar cargas (motores, relés) con control por voltaje, ideal en controladores de potencia en proyectos Arduino.

398. BJT (transistor bipolar)

Dispositivo semiconductor que puede amplificar corriente; aunque hoy muchas aplicaciones usan MOSFETs, aprender su operación es útil para fundamentos.

399. Relé (módulo)

Interruptor controlado eléctricamente que permite encender/apagar cargas de mayor voltaje usando una señal de baja potencia desde Arduino.

400. Driver de motor (L298, L293D)

Circuito que permite controlar dirección y velocidad de motores DC y paso a paso desde microcontroladores sin dañar pines de salida.

401. Motor paso a paso

Motor que avanza en pasos discretos permitiendo control de posición preciso; común en impresoras 3D y robots.

402. Servo motor

Motor con control por ángulo (posicionamiento) común en robótica educativa; recibe señales PWM para situarse en una posición deseada.

403. PWM (Pulse Width Modulation)

Técnica que simula niveles analógicos variando el tiempo de encendido/apagado de una señal digital; se usa para controlar velocidad de motores y brillo de LEDs.

404. ADC (Analog to Digital Converter)

Convertidor que transforma una señal analógica (voltaje) en un valor digital que el microcontrolador puede procesar.

405. DAC (Digital to Analog Converter)

Convierte valores digitales en señales analógicas; menos común en microcontroladores básicos, pero útil en audio y control analógico.

406. Debounce (antirrebote)

Técnica para filtrar señales erráticas de botones mecánicos, evitando lecturas múltiples por un solo pulso físico.

407. I2C (Integrated Circuit)

Protocolo de comunicación serial para conectar múltiples dispositivos usando solo dos líneas (SDA y SCL); común en sensores y pantallas.

408. SPI (Serial Peripheral Interface)

Protocolo serial de alta velocidad con líneas MOSI/MISO/SCLK/SS para comunicarse con periféricos como memorias y convertidores.

409. UART / Serial (RX/TX)

Comunicación serie punto a punto usada para depuración, módulos GPS o comunicación entre microcontroladores; sencilla pero importante en prototipos.

410. I/O digital vs analógico (resumen)

Digitales: dos estados (0/1); analógicas: rango continuo. Elegir según el sensor o actuador que vas a leer o controlar.

411. Watchdog timer

Temporizador que reinicia el microcontrolador si el software queda bloqueado; técnica simple para mejorar robustez en sistemas embebidos.

412. Bootloader (microcontrolador)

Programa pequeño que permite cargar nuevo firmware sin programador externo; Arduino usa bootloader para facilitar la programación vía USB.

413. Firmware

Software específico que corre dentro de un dispositivo (microcontrolador, placa); controlar versiones y actualizaciones del firmware es esencial para estabilidad.

414. EEPROM (memoria no volátil)

Memoria que conserva datos al apagar el dispositivo, útil para almacenar configuraciones y calibraciones en proyectos.

415. Flash memory

Tipo de memoria no volátil usada para almacenar firmware y datos; en microcontroladores y tarjetas SD es la base del almacenamiento persistente.

416. Hysteresis (en sensores)

Rango de tolerancia alrededor de un umbral para evitar oscilaciones rápidas en salidas cuando la señal está cerca del punto de corte.

417. Calibration (calibración)

Ajuste de sensor o sistema para corregir desviaciones y obtener lecturas más cercanas a la realidad; imprescindible en mediciones precisas.

418. Sampling rate / Frecuencia de muestreo

Número de lecturas por segundo que se toman de un sensor; influye en precisión temporal y en la posibilidad de detectar eventos rápidos.

419. Aliasing (aliasing)

Fenómeno donde señales muestreadas a baja frecuencia parecen diferentes (se pliegan), evitarlo con muestreo adecuado (Nyquist) y filtros.

420. Matriz de LEDs (LED matrix)

Conjunto de LEDs organizados en filas/columnas controlados por multiplexación para mostrar información o gráficos simples.

421. Pantalla OLED / TFT

Pantallas con mayor resolución y color que LCDs básicos; permiten interfaces gráficas ricas en proyectos embebidos.

422. RTOS (Real Time Operating System):

Sistema operativo de tiempo real que gestiona tareas con restricciones temporales; útil para proyectos que requieren respuesta determinista.

423. Task scheduling (planificación de tareas)

Organizar ejecución de funciones periódicas o event-driven para evitar bloqueos y garantizar respuesta en sistemas embebidos.

424. Interrupt (interrupción)

Mecanismo que permite al hardware interrumpir la ejecución normal para atender eventos prioritarios (botón, sensor), mejorando eficiencia.

425. Polling (lectura por sondeo)

Método donde el microcontrolador consulta periódicamente el estado de un dispositivo; más simple pero menos eficiente que interrupciones.

426. Serial Monitor (IDE)

Herramienta que permite ver mensajes desde Arduino/MCU para depuración, registros de sensores o estado de ejecución en tiempo real.

427. OTA (Over The Air) updates

Actualizaciones de firmware por red (Wi-Fi) sin necesidad de conectar físicamente el dispositivo; comodidad que exige medidas de seguridad.

428. MQTT broker (ej. Mosquitto)

Servidor que recibe, distribuye y gestiona mensajes MQTT entre dispositivos; actúa como intermediario en arquitecturas publish/subscribe.

429. REST (Representational State Transfer)

Estilo arquitectónico para APIs web usando HTTP y recursos identificables por URL, promueve simplicidad y uso de métodos estándar.

430. CRUD (Create, Read, Update, Delete)

Operaciones básicas para gestionar recursos en aplicaciones y bases de datos; forman la base de la mayoría de aplicaciones web.

431. Stateless (sin estado)

Servidor que no guarda información de la sesión entre peticiones; simplifica escalado pero obliga al cliente o tokens a llevar la información de estado.

432. Stateful (con estado)

Servidor o sistema que mantiene información entre peticiones (sesiones), puede facilitar interacciones continuas pero complica escalado.

433. Token (autenticación)

Credencial digital (ej. JWT) que representa identidad/privilegios y se usa para autorizar peticiones sin enviar credenciales cada vez.

434. JWT (JSON Web Token)

Formato de token compacto que incluye cabecera, payload y firma; permite verificar integridad y autenticidad sin consultar sesión en servidor en cada petición.

435. OAuth (authorization framework)

Sistema que delega acceso a recursos sin compartir credenciales; común para logins con Google/Facebook y acceso a APIs en nombre del usuario.

436. HTTPS / TLS (seguridad)

HTTPS cifra tráfico HTTP mediante TLS/SSL para proteger confidencialidad e integridad; certificados y configuración segura son críticos para confianza en la web.

437. CA (Certificate Authority)

Entidad que emite certificados digitales que validan la identidad de sitios web; los navegadores confían en CAs para establecer conexiones seguras.

438. Hashing (resumen)

Técnica que transforma datos a una huella (hash) de longitud fija; útil para almacenar contraseñas de forma segura (con sal) y verificar integridad.

439. Salting (sal)

Añadir datos aleatorios a contraseñas antes de hashearlas para evitar ataques por diccionario y uso de tablas rainbow.

440. RSA (criptografía asimétrica)

Algoritmo de cifrado y firma digital que usa pares de claves pública/privada; más lento que cifrado simétrico pero permite intercambio seguro de claves.

441. AES (Advanced Encryption Standard)

Algoritmo de cifrado simétrico ampliamente usado por su eficiencia y seguridad en comunicaciones y almacenamiento cifrado.

442. VPN tipo site to-site vs remote-access

Site-to-site conecta redes completas entre sí; remote-access conecta dispositivos individuales a una red remota con acceso protegido.

443. Penetration Testing (Pentest)

Práctica controlada de atacar sistemas para descubrir vulnerabilidades antes de que lo hagan actores maliciosos; produce reportes y recomendaciones.

444. Hardening (endurecimiento)

Conjunto de acciones para reducir la superficie de ataque (actualizar, deshabilitar servicios, configurar firewalls), mejorando la seguridad de sistemas.

445. Programación Orientada a Aspectos (AOP)

Metodología de programación que busca aumentar la modularidad separando las preocupaciones transversales (cross-cutting concerns), como el logging, la seguridad o la gestión de transacciones, de la lógica de negocio principal. Esto se logra inyectando (weaving) código (aspectos) en puntos específicos de la ejecución (join points).

446. RAID 0/1/5/10 (conceptos básicos)

Conjuntos de discos organizados para mejorar rendimiento (striping) o redundancia (mirroring/parity); elegir según necesidad de velocidad y tolerancia a fallos.

447. Filesystem (sistema de archivos)

Estructura y formato que organiza cómo se almacenan y encuentran archivos en un disco (FAT32, NTFS, ext4); elegir según compatibilidad y características.

448. Partition (partición)

División lógica de un disco en secciones que pueden albergar diferentes archivos o sistemas operativos; facilita organización y backups.

449. Mount (montar un sistema de archivos)

Proceso de hacer accesible un sistema de archivos en una ruta del árbol global; en Linux se usa mount/umount.

450. Format (formatear)

Preparar un disco con un sistema de archivos; formatear borrará datos, por lo que debe hacerse con cuidado y con backup previo.

451. Throughput vs Latency

Throughput = cantidad de datos transferidos por unidad de tiempo; Latency = tiempo de demora en una operación. Ambos influyen en experiencia de usuario y rendimiento.

452. Content Delivery Network (CDN)

Red de servidores distribuidos que cachean contenido estático (imágenes, JS) para entregar recursos más cerca del usuario y reducir latencia.

453. Load balancer (balanceador de carga)

Distribuye tráfico entre múltiples servidores para mejorar disponibilidad y rendimiento; puede ser por round-robin, least-connections o basado en salud.

454. Auto scaling (escalado automático)

Mecanismo que añade o quita instancias (servidores) según demanda, optimizando recursos y coste en entornos en la nube.

455. Container vs Virtual Machine (VM)

Contenedor empaqueta app y dependencias compartiendo kernel; VM virtualiza hardware y corre un OS completo; contenedores son más ligeros y rápidos de arrancar.

456. Image (Docker)

Plantilla inmutable que contiene la aplicación y sus dependencias; a partir de una imagen se crean contenedores ejecutables.

457. Volume (Docker)

Mecanismo para persistir datos fuera del ciclo de vida del contenedor, necesario para conservar información tras reinicios o recreaciones.

458. CI/CD (Integración Continua / Despliegue Continuo)

Prácticas y pipelines que automatizan pruebas, builds y despliegues para acelerar entrega con calidad y permitir rollbacks controlados.

459. TDD (Test Driven Development)

Práctica donde se escriben tests antes del código funcional; guía el diseño y asegura que cada nueva función está cubierta por pruebas.

460. Refactor (refactorización)

Mejorar la estructura interna del código sin cambiar su comportamiento observable; mantiene base de código limpia y fácil de mantener.

461. Design pattern (patrón de diseño)

Solución probada a un problema común de diseño de software (ej. Singleton, Factory, Observer); ayudan a crear código más organizado y comprensible.

462. MVC (Model View-Controller)

Patrón de arquitectura que separa la lógica de datos (Model), presentación (View) y control de flujo (Controller); mejora mantenibilidad en aplicaciones.

463. MVVM (Model View-ViewModel)

Variante arquitectónica popular en entornos con binding de datos (ej. aplicaciones de escritorio/móviles) para separar responsabilidades.

464. RESTful API (principios)

APIs que siguen principios REST: recursos identificables por URL, uso de métodos HTTP y representación por formatos uniformes (JSON).

465. SOAP (Simple Object Access Protocol)

Protocolo XML para intercambio estructurado en servicios web; más verboso que REST y con estándares de seguridad integrados (WS-Security).

466. GraphQL (consulta flexible)

Lenguaje de consulta para APIs que permite al cliente pedir exactamente los campos necesarios, reduciendo overfetching y underfetching.

467. CORS (Cross Origin Resource Sharing)

Política de seguridad del navegador que controla cómo las páginas web pueden hacer peticiones a dominios distintos; requiere configuración en servidores para permitir accesos.

468. Same Origin Policy

Restricción del navegador que evita que scripts de un origen accedan a recursos de otro sin permiso; base de CORS y seguridad web.

469. CSV (Comma Separated Values)

Formato sencillo para datos tabulares; fácil de generar y procesar en hojas de cálculo y scripts.

470. Base de datos relacional (RDBMS)

Sistema que organiza datos en tablas con relaciones y usa SQL para consultas; ideal para datos estructurados y transaccionales.

471. Diseño de Zero-Knowledge Proofs (ZKP) Aplicados

Protocolos criptográficos que permiten a una parte (el probador) demostrar a otra (el verificador) que una declaración es verdadera, sin revelar ninguna información sobre la declaración en sí.

472. Diseño de Sistemas Inmutables (Immutable Systems)

Paradigma de diseño donde los componentes de infraestructura (servidores, contenedores, máquinas virtuales) nunca se modifican una vez desplegados. En lugar de aplicar un parche o una actualización a un servidor existente, se destruye el antiguo y se despliega uno completamente nuevo con la configuración deseada.

473. Automatización del Descubrimiento de Deuda Técnica (Technical Debt Mapping)

Uso de herramientas de análisis estático avanzado, métricas de complejidad ciclomática y técnicas de code profiling para medir, mapear y priorizar la deuda técnica automáticamente a través de un gran codebase. El objetivo es transformar la deuda de un concepto abstracto a una métrica cuantificable y gestionable.

474. Normalization (normalización)

Proceso de estructurar una base de datos para reducir redundancia y mejorar integridad (formas normales: 1NF, 2NF, 3NF).

475. Denormalization (desnormalización)

Decisión deliberada de duplicar datos para mejorar rendimiento de lectura en sistemas de alta carga; requiere cuidado para mantener consistencia.

476. Index (índice en BD)

Estructura que acelera consultas en columnas específicas; elegir índices adecuados mejora rendimiento pero consume espacio y afecta inserciones.

477. Transaction (transacción)

Conjunto de operaciones que se ejecutan de forma atómica; principios ACID (Atomicidad, Consistencia, Aislamiento, Durabilidad) garantizan integridad.

478. ACID vs BASE (consistencia)

ACID para transacciones fuertes en RDBMS; BASE (Basically Available, Soft state, Eventual consistency) describe modelos más flexibles en sistemas distribuidos.

479. Sharding (fragmentación)

Dividir datos en múltiples servidores para escalar horizontalmente; aumenta complejidad de consultas y gestión.

480. Replication (replicación)

Copia de datos entre servidores para alta disponibilidad y tolerancia a fallos; puede ser sincrónica o asincrónica.

481. ORM (Object Relational Mapping)

Herramienta que mapea objetos de un lenguaje (clases) a tablas de una base de datos, facilitando acceso y mantenimiento del código.

482. Diseño y Gestión de Application Binary Interfaces (ABI)

Estándares de bajo nivel que definen cómo el código compilado (binario) interactúa con el sistema operativo y con otras librerías. A diferencia de una API (Application Programming Interface), que es para humanos (código fuente), la ABI es para máquinas y define cómo se llaman las funciones, cómo se pasan los argumentos en la pila y el formato de los datos binarios.

483. Cross Site Scripting (XSS)

Vulnerabilidad donde scripts maliciosos se inyectan en páginas web; prevenir con escaping, validación y políticas de contenido.

484. CSRF (Cross Site Request Forgery)

Ataque que induce a un usuario autenticado a ejecutar acciones no deseadas; mitigación con tokens anti-CSRF y cookies seguras.

485. OAuth 2.0 (flujos)

Provee varios flujos (authorization code, implicit, client credentials) para distintos escenarios de autorización entre aplicaciones.

486. Pruebas Fuzzing y Pruebas de Mutación

Formas avanzadas de testing para descubrir fallos de seguridad o bugs catastróficos inyectando datos de entrada aleatorios, malformados o inesperados (fuzzing) o mutando el código existente (mutación).

487. SAML (Security Assertion Markup Language)

Estándar XML para intercambiar información de autenticación/atributos entre IdP y SP, común en entornos corporativos SSO.

488. OpenID Connect

Capa de identidad construida sobre OAuth 2.0 que permite autenticación federada y obtiene información del usuario de forma estandarizada.

489. Cómputo de Streams con Ventanas (Time-Windowing)

Técnica esencial en el procesamiento de Big Data en tiempo real que consiste en agrupar flujos continuos e ilimitados de datos (streams) en segmentos lógicos (ventanas) basados en el tiempo o el conteo de eventos. Permite calcular métricas (e.g., el promedio de clics en los últimos 5 minutos) sobre datos en movimiento.

490. WAF (Web Application Firewall)

Filtra y protege aplicaciones web de ataques comunes analizando tráfico HTTP y aplicando reglas o aprendizaje automático.

491. IDS/IPS (detección / prevención de intrusos)

Sistemas que detectan (IDS) o bloquean (IPS) actividades sospechosas en la red; forman parte de una estrategia de defensa en profundidad.

492. SIEM (Security Information and Event Management)

493. Endpoint protection (antimalware)

Conjunto de herramientas (antivirus, EDR) que protegen dispositivos finales contra malware y actividades sospechosas. 500. Encryption at rest vs in transit: Cifrado at-rest protege datos almacenados; in-transit protege datos en movimiento (TLS); ambos son prácticas recomendadas para seguridad de datos. 501. PKI (Public Key Infrastructure): Conjunto de políticas, autoridades y herramientas que gestionan claves públicas/privadas, certificados y confianza en comunicaciones seguras.

494. Diseño de Lenguajes Específicos del Dominio (DSL)

Creación de un lenguaje de programación pequeño y altamente especializado diseñado para resolver un único conjunto de problemas o representar una lógica de negocio compleja de manera concisa y no ambigua. Los DSLs pueden ser internos (construidos dentro de un lenguaje como Python) o externos (con su propio parser).

495. Data retention policy (política de retención)

Normas que definen cuánto tiempo conservar datos por requisitos legales o de negocio y cuándo eliminarlos para protección de privacidad.

496. GDPR (protección de datos contexto)

Reglamento europeo que establece derechos y obligaciones sobre tratamiento de datos personales; su conocimiento es importante para proyectos que manejan datos de usuarios.

497. Privacy by Design

Enfoque que integra privacidad y protección de datos desde la fase de diseño de sistemas y productos, no como un añadido posterior.

498. Anonymization vs Pseudonymization

Anonimizar elimina identificadores irreversiblemente; pseudonimizar sustituye identificadores pero permite reversión controlada con llave.

499. Data pipeline (tubería de datos)

Flujo de extracción, transformación y carga (ETL/ELT) que lleva datos desde fuentes hasta almacenes para análisis y consumo.

500. ETL (Extract, Transform, Load)

Proceso clásico para integrar y preparar datos en un almacén o base para análisis: extraer origen, transformar a formato útil, cargar destino.

501. OLAP vs OLTP

OLAP para análisis y consultas complejas (reporting), OLTP para transacciones cortas y frecuentes (sistemas operativos).

502. Business Intelligence (BI)

Conjunto de técnicas y herramientas para analizar datos y generar información útil para la toma de decisiones (dashboards, reporting).

503. Data visualization (visualización)

Representar datos gráficamente (barras, líneas, mapas) para facilitar la comprensión y la toma de decisiones; buenas visualizaciones priorizan claridad y contexto.

504. CSV import/export (importación/ exportación)

Práctica común para mover datos entre sistemas; cuidar codificación, delimitadores y formatos de fecha para evitar errores.

505. Regular expressions (regex)

Herramienta potente para buscar y manipular texto con patrones; útil en validaciones, parsing y tareas de transformación.

506. Parsing (análisis sintáctico)

Proceso de interpretar texto estructurado (JSON, CSV, logs) para convertirlo en estructuras de datos que un programa puede usar.

507. Ingeniería de Lanzamiento con Estrategias Avanzadas (Progressive Delivery)

Conjunto de técnicas para desplegar nuevas funcionalidades de manera incremental y controlada para un subconjunto de usuarios, minimizando el riesgo de fallos a gran escala

508. NLP (Natural Language Processing)

Área de IA que trabaja con lenguaje humano para tareas como clasificación, extracción de entidades y generación de texto.

509. Arquitectura de Patrones Saga (Gestión de Transacciones Distribuidas)

Patrón para manejar transacciones de negocio que abarcan múltiples microservicios (o bases de datos), donde no se puede usar una transacción atómica tradicional (ACID).

510. Stemming / Lemmatization

Técnicas para reducir palabras a su raíz (stem) o forma canónica (lemma) para normalizar texto antes de análisis.

511. Bag of Words (BoW)

Representación simple de texto como conteo de palabras; método básico para convertir texto a vectores numéricos.

512. TF IDF (Term Frequency_Inverse Document Frequency)

Métrica que pondera la importancia de una palabra en un documento respecto a un corpus; útil en búsquedas y clasificación.

513. Embedding (representaciones vectoriales)

Representación numérica densa de palabras o frases que captura similitudes semánticas; base para muchos modelos modernos de NLP.

514. Supervised vs Unsupervised learning

Supervised: modelos que aprenden de datos con etiquetas; Unsupervised: descubren patrones sin etiquetas (clusterización).

515. Patrón Sidecar (Contenedores Auxiliares)

Patrón de diseño de contenedores que adjunta un contenedor auxiliar (el sidecar) a un contenedor principal de la aplicación para extender sus funcionalidades, como la observabilidad, la seguridad o la red, sin modificar el código de la aplicación.

516. Modelo Circuit Breaker (Disyuntor)

Patrón de diseño de software para evitar que una falla en un servicio secundario (dependencia) paralice todo el sistema. Funciona como un fusible eléctrico: si un servicio dependiente comienza a fallar repetidamente, el Circuit Breaker detiene las llamadas a ese servicio temporalmente, permitiéndole recuperarse y previniendo la degradación en cascada.

517. Metaprogramación y Generación de Código en Tiempo de Compilación

Técnica avanzada donde un programa escribe o manipula otro programa (o a sí mismo) para reducir el código repetitivo (boilerplate) o implementar lógica compleja de manera concisa. El enfoque más maduro ocurre en tiempo de compilación para no incurrir en costos de rendimiento en tiempo de ejecución.

518.Ingeniería de Resiliencia (Chaos Engineering)

La disciplina de experimentar en un sistema distribuido en producción para construir confianza en la capacidad del sistema para resistir condiciones turbulentas e inesperadas. Implica la inyección deliberada y controlada de fallos (ej. killing de servidores, latencia de red) para encontrar y corregir debilidades antes de que causen interrupciones reales.

519. Ingeniería de la Confiabilidad Humana (HRE)

La aplicación de principios de Ingeniería de Fiabilidad (SRE) a los procesos y la cultura de los equipos, enfocándose en cómo el diseño de sistemas puede reducir la carga cognitiva, el estrés y el error humano en las operaciones diarias.

520. Principio de Mínimo Privilegio (PoLP)

Principio fundamental de seguridad que establece que a cualquier usuario, programa o proceso se le deben conceder solo los permisos y recursos que son estrictamente necesarios para realizar su función designada, y nada más.

521. Dropout (técnica en redes neuronales)

Durante entrenamiento, apaga aleatoriamente neuronas para evitar co-dependencias y mejorar generalización del modelo.

522. Batch vs Online learning

Batch: entrenar con todo el dataset; Online: actualizar el modelo continuamente con nuevas muestras (útil para datos streaming).

523. I²C / TWI

Protocolo de comunicación serial de dos cables (SDA y SCL) muy popular en Arduino para que el microcontrolador hable con múltiples sensores o pantallas usando solo dos pines.

524. Zero Trust Architecture (Arquitectura de Confianza Cero)

Modelo de seguridad que elimina la presunción implícita de confianza que se le da a cualquier usuario o dispositivo dentro de un perímetro de red (el enfoque de 'confiar, pero verificar'). En su lugar, todo intento de acceso debe ser autenticado y autorizado continuamente, independientemente de la ubicación de origen.

525. Data Mesh (Malla de Datos)

Paradigma de arquitectura de datos descentralizada que trata a los datos como un producto, delegando la propiedad y la responsabilidad de los conjuntos de datos a los equipos de dominio que los producen (en lugar de tener un equipo centralizado de data warehouse).

526. Patrón Service Mesh (Malla de Servicios)

Una capa de infraestructura dedicada a manejar las comunicaciones servicio-a-servicio en arquitecturas de microservicios. Proporciona características como el descubrimiento de servicios, balanceo de carga, cifrado TLS mutuo, y métricas de tráfico sin necesidad de programar estas funcionalidades en cada servicio

527. Refactorización de Código (Refactoring) para Deuda Técnica

Proceso disciplinado de reestructurar el código existente sin cambiar su comportamiento externo, con el único objetivo de mejorar sus cualidades no funcionales (legibilidad, mantenibilidad y complejidad estructural, o Deuda Técnica).

528. Data augmentation

Técnica para crear más datos artificiales (rotaciones, ruido) y mejorar robustez de modelos de visión u otros dominios.

529. Computer vision (visión por computador)

Área de IA que interpreta imágenes y video para detectar, clasificar y segmentar objetos; útil en robótica y aplicaciones de seguridad.

530. Acoplamiento y Cohesión (Métricas de Diseño de Software)

Dos métricas fundamentales para medir la calidad estructural del Software. Cohesión es la medida de cuán relacionadas están las responsabilidades de los elementos dentro de un mismo módulo (se desea alta cohesión). Acoplamiento es la medida de cuán dependiente es un módulo de otros (se desea bajo acoplamiento).

531. Segmentation (segmentación)

Técnica que asigna etiqueta a cada píxel de una imagen para delimitar objetos de forma precisa (ej. segmentación semántica).

532. Transfer learning (aprendizaje por transferencia)

Reutilizar modelos preentrenados como base y ajustarlos a una tarea específica para ahorrar datos y tiempo.

533. Preprocessing (preprocesamiento)

Pasos para limpiar y transformar datos (normalizar, imputar valores) antes de entrenar modelos, crucial para buenos resultados.

534. Feature engineering (ingeniería de características)

Crear o transformar variables que capturan mejor la relación entre entrada y salida para mejorar modelos.

535. One hot encoding

Técnica para convertir variables categóricas en vectores binarios, preparándolas para algoritmos que requieren entradas numéricas.

536.Patrón Event Storming (Tormenta de Eventos)

Taller colaborativo y de ritmo rápido que utiliza notas adhesivas de colores para mapear visualmente todo el flujo de trabajo de un dominio de negocio complejo, identificando los "eventos" clave (lo que sucede) que impulsan el sistema. Es una técnica de Modelado de Dominio (DDD) y gestión de requisitos.

537. Scope (Ámbito o Alcance de Variables)

La región del código donde una Variable puede ser accedida o referenciada. El scope define la visibilidad y el tiempo de vida de la variable. Los tipos comunes son Global (accesible en todo el programa) y Local (accesible solo dentro de una función o bloque de código).

538. GPIO (General Purpose Input/Output)

Los pines más básicos en el Hardware del microcontrolador (como Arduino) que pueden ser configurados por Software para funcionar como una entrada digital (leer un botón) o una salida digital (encender un LED).

539. Hierarchical clustering

Método de clusterización que crea árbol de grupos (dendrograma) y permite explorar relaciones a diferentes niveles.

540. Anomaly detection (detección de anomalías)

Identificación de eventos o registros atípicos que pueden indicar fallos, fraudes o comportamientos raros.

541. Time series (series temporales)

Datos ordenados por tiempo; requieren técnicas y métricas específicas (ARIMA, smoothing) para modelar tendencias y estacionalidad.

542. Tasa de Refresco (Refresh Rate) (Hz)

La frecuencia con la que una pantalla (o un proyector) actualiza la imagen que muestra, medida en Hertz (Hz). Una tasa de refresco más alta (ej., 120 Hz) significa una imagen más fluida y menos fatiga visual.

543. Modelo OSI (Open Systems Interconnection Model)

Un marco conceptual que divide las funciones de la comunicación en red en siete capas abstractas. Se utiliza para entender y estandarizar cómo los diferentes Protocolos y tecnologías interactúan, desde la capa física (Hardware) hasta la capa de aplicación (HTTP).

544. Costo de Swap (Intercambio de Memoria)

El rendimiento lento y degradado que ocurre cuando el sistema operativo necesita mover datos poco usados desde la RAM rápida a una zona de almacenamiento más lenta (disco HDD/SSD) o swap file para liberar espacio en la RAM. Este proceso de intercambio es muy costoso en tiempo.

545. Monitoring (monitoreo)

Practica de recopilar métricas y logs (CPU, memoria, latencia) para vigilar salud de sistemas, detectar incidentes y actuar con rapidez.

546. Alertas en sistemas

Notificaciones automáticas que informan al usuario o administrador cuando ocurre un evento importante, como un error, sobrecarga o fallo en un servicio.

547. Log (registro)

Archivo donde un sistema o aplicación guarda eventos ocurridos (errores, accesos, procesos), útil para monitoreo y resolución de problemas.

548. Bitácora digital

Conjunto de registros ordenados que muestran la historia de actividad en un sistema, similar a un diario pero para eventos informáticos.

549. Panel de control

Interfaz central donde se configuran parámetros de hardware, software o red de un sistema operativo o aplicación.

550. Configuración por defecto

Ajustes que trae un programa o dispositivo al instalarse; suelen ser funcionales, pero es recomendable personalizarlos para seguridad y eficiencia.

551. Usuario administrador

Cuenta con permisos completos sobre un sistema, capaz de instalar programas, cambiar configuraciones y gestionar usuarios.

552. Usuario estándar

Cuenta limitada que permite usar aplicaciones pero no realizar cambios críticos en el sistema, mejorando la seguridad.

553. Permisos de archivo

Reglas que determinan quién puede leer, modificar o ejecutar un archivo o carpeta en un sistema.

554. Sistema multiusuario

Entorno que permite que varios usuarios trabajen en el mismo sistema sin interferir entre sí, como en servidores.

555. Sistema monousuario

Diseñado para un solo usuario a la vez, común en computadoras personales antiguas o sistemas sencillos.

556. Procesador multinúcleo

CPU con varios núcleos de ejecución que permiten realizar tareas en paralelo, mejorando la velocidad en programas modernos.

557. Hilo (thread)

Unidad mínima de ejecución dentro de un programa, que permite realizar tareas concurrentes.

558. Serialización y Deserialización (De Datos)

Serialización es el proceso de convertir un Objeto de Software complejo (como una Variable o un conjunto de Datos) en un formato de bytes o texto simple y secuencial (ej., JSON o XML) para poder enviarlo a través de Internet o almacenarlo en un archivo. Deserialización es el proceso inverso.

559. Planificador de procesos

Componente del sistema operativo que decide qué proceso se ejecuta y por cuánto tiempo.

560. Memoria virtual

Técnica que usa espacio del disco como extensión de la memoria RAM para ejecutar programas más grandes.

561. Archivo de paginación

Parte del disco usada por la memoria virtual; cuando la RAM se llena, el sistema mueve allí datos temporales.

562. Gestión de memoria

Conjunto de técnicas que el sistema operativo usa para asignar, liberar y organizar memoria a procesos.

563. Cola de impresión

Lista de documentos en espera para ser impresos, que se gestionan en orden de llegada o prioridad.

564. Controlador de impresora

Software que traduce las órdenes del sistema a instrucciones que entiende la impresora.

565. Impresora de inyección de tinta

Dispositivo que imprime rociando tinta líquida sobre el papel, ideal para documentos y fotografías.

566. Impresora láser

Usa tóner en polvo y luz láser para imprimir rápidamente con alta calidad, común en oficinas.

567. Escáner

Dispositivo que convierte documentos físicos en archivos digitales de imagen o texto.

568. Digitalización de documentos

Proceso de convertir papeles físicos en archivos digitales mediante escáner o cámara.

569. OCR (Reconocimiento Óptico de Caracteres)

Tecnología que permite convertir imágenes escaneadas de texto en texto editable.

570. PDF (Portable Document Format)

Formato de archivo que conserva diseño, fuentes e imágenes, ideal para compartir documentos.

571. Compresión de archivos

Proceso de reducir el tamaño de un archivo para ahorrar espacio o facilitar su envío.

572. Archivos ZIP/RAR

Formatos de compresión que permiten almacenar varios archivos juntos en un paquete más pequeño.

573. Descompresión

Proceso de restaurar archivos comprimidos a su estado original.

574. Unidad óptica

Dispositivo que lee o graba discos CD, DVD o Blu-ray, cada vez menos usado por almacenamiento en la nube.

575. USB (Universal Serial Bus)

Estándar de conexión que permite conectar dispositivos como memorias, teclados o impresoras.

576. Cuello de Botella (Bottleneck) de Rendimiento

El componente o paso más lento de un sistema que, debido a su capacidad limitada, restringe la velocidad general del flujo de trabajo o la aplicación. En Hardware, podría ser la CPU o la RAM; en Software, podría ser una consulta lenta a la base de datos o un Punto Final saturado.

577. Sistema de Detección de Colisiones (Collision Detection)

Un algoritmo en la Lógica de Programación que determina si dos o más elementos (objetos, imágenes, coordenadas) en un entorno simulado (como un videojuego o una interfaz) se superponen o entran en contacto físico.

578. Web Scraping (Raspado de Datos Web)

Técnica de Software para extraer automáticamente grandes cantidades de Datos de sitios Web mediante el uso de programas o bots. Esto se hace analizando el código HTML de las páginas web (el HyperText).

579. Puertos de Red (Network Ports)

Un número (del 0 al 65535) que se adjunta a una Dirección IP para identificar qué aplicación o servicio específico debe recibir los Datos en un servidor. Por ejemplo, el puerto 80 es el estándar para HTTP y el 443 para HTTPS.

580. Fugas de Memoria (Memory Leaks)

Un error en la Lógica de Programación que ocurre cuando el Software reserva espacio en la RAM para una Variable o un objeto, pero luego olvida liberarlo después de que ya no es necesario. Con el tiempo, esto agota la RAM disponible y puede causar que la aplicación se congele o colapse.

581. Memoria NVMe

Tecnología de almacenamiento aún más rápida que los SSD comunes, conectada directamente a la placa base.

582. Codificación de Caracteres UTF-8 (Unicode)

El sistema de codificación de caracteres más común para el texto en la Web y el Software. UTF-8 puede representar prácticamente todos los caracteres y símbolos del mundo (letras latinas, chinas, emojis, etc.) de manera eficiente, usando de 1 a 4 bytes por carácter.

583. Principio de YAGNI (You Aren't Gonna Need It)

Un principio fundamental de la Gestión de Proyectos y Scrum que sugiere que nunca se debe añadir funcionalidad a un Software hasta que sea absolutamente necesario. Esto evita la sobreingeniería y reduce la deuda técnica y el Alcance innecesario.

584. Google Drive

Plataforma de almacenamiento en la nube con integración a herramientas de oficina de Google.

585. OneDrive

Servicio de almacenamiento en la nube de Microsoft integrado en Windows y Office.

586. Dropbox

Plataforma de almacenamiento en la nube que permite sincronización de archivos entre dispositivos.

587. Sincronización

Proceso que mantiene iguales los archivos en varios dispositivos conectados a la misma cuenta.

588. Arquitectura Client-Side vs. Server-Side (Procesamiento Web)

Dos modelos que definen dónde ocurre la mayor parte del trabajo. En Client-Side (lado del cliente, ej. tu navegador), el código Software (JavaScript) se ejecuta principalmente en el dispositivo del usuario, aliviando la carga del Hosting. En Server-Side (lado del servidor), el servidor hace todo el trabajo y envía el resultado final.

589. Frecuencia de Reloj (Clock Speed) (Hz)

La velocidad a la que el CPU ejecuta los ciclos de procesamiento de sus instrucciones, medida en Hertz (Hz), MegaHertz (MHz) o GigaHertz (GHz). Un ciclo de reloj es el tiempo que tarda la CPU en realizar una operación básica.

590. Código de Máquina (Machine Code)

El lenguaje de programación de más bajo nivel posible, compuesto enteramente por instrucciones Binarias (secuencias de 1s y 0s) que son *directamente ejecutables por el CPU sin necesidad de ser interpretadas o traducidas. Es el resultado final de la compilación.

591. Restauración de sistema

Función que devuelve un sistema a un punto anterior, útil para resolver fallos.

592. Punto de restauración

Copia del estado de configuraciones y archivos clave que permite regresar a un momento previo. 601. Clonación de disco: Crear una copia exacta de un disco duro, incluyendo sistema operativo y programas.

593. Antivirus

Programa que detecta y elimina software malicioso, protegiendo el sistema de infecciones.

594. Antimalware

Similar al antivirus, pero con mayor capacidad para detectar programas dañinos modernos como spyware o ransomware.

595. Spyware

Programa que espía la actividad del usuario sin permiso, recolectando datos privados.

596. Adware

Software que muestra anuncios no deseados en un sistema, a veces junto con spyware.

597. Ransomware

Malware que bloquea los archivos y pide un rescate para liberarlos.

598. Keylogger

Programa que registra las teclas pulsadas, usado maliciosamente para robar contraseñas.

599. Phishing

Técnica de engaño para obtener información confidencial haciéndose pasar por empresas legítimas.

600. Smishing

Variante del phishing que llega mediante mensajes SMS con enlaces falsos.

601. Vishing

Estafa que busca robar información por llamadas telefónicas.

602. Malware

Término general para referirse a programas dañinos como virus, gusanos y troyanos.

603. Virus informático

Programa que se copia a otros archivos y puede dañar datos o ralentizar sistemas.

604. Gusano (worm)

Malware que se replica automáticamente a través de redes, consumiendo recursos.

605. Troyano

Programa que aparenta ser legítimo pero da acceso a atacantes al sistema infectado.

606. Botnet

Red de computadoras infectadas que son controladas a distancia para ataques masivos.

607. Ataque DDoS

Saturación de un servicio en Internet mediante miles de peticiones simultáneas desde una botnet.

608. Cortafuegos de hardware

Dispositivo dedicado que protege una red completa frente a accesos externos.

609. Cortafuegos de software

Aplicación instalada en un sistema operativo que protege al equipo individual.

610. Política de contraseñas

Conjunto de reglas que determinan la complejidad y caducidad de claves en un sistema.

611. Autorización

Proceso que define qué recursos puede usar un usuario una vez autenticado.

612. Autenticación de dos factores (2FA)

Método de seguridad que combina contraseña más otro factor, como SMS o app de verificación.

613. Formato Hexadecimal (Base 16)

Un sistema de numeración que utiliza la base 16 (dígitos del 0 al 9 y letras de la A a la F) para representar números. Es usado extensamente en la programación y el Hardware como una forma legible de representar números Binarios muy largos (cada dígito hexadecimal representa exactamente 4 bits).

614. Certificado digital

Documento electrónico emitido por una entidad de confianza que garantiza la identidad de un sitio o usuario.

615. Algoritmo Hash y Función de Resumen (Digest)

Una función matemática unidireccional que toma cualquier cantidad de Datos (texto, archivo, Variable) y produce una salida de longitud fija (el hash o resumen) que es única para esa entrada. Se utiliza para verificar la integridad de los Datos (si un archivo ha sido alterado) o para almacenar contraseñas de forma segura.

616. Proxy inverso

Servidor que recibe peticiones externas y las distribuye a varios servidores internos, protegiendo la infraestructura.

617. Balanceador de carga

Dispositivo o software que reparte tráfico entre varios servidores para mejorar rendimiento.

618. Latencia

Tiempo que tarda un paquete de datos en viajar desde el origen hasta el destino.

619. Jitter

Variación en el tiempo de llegada de paquetes, afecta la calidad de llamadas y videoconferencias.

620. Packet loss (pérdida de paquetes)

Cuando algunos datos no llegan a destino, causando cortes en conexiones.

621. Protocolo TCP

Garantiza que los paquetes lleguen completos y en orden, ideal para navegación web.

622. Protocolo UDP

No garantiza entrega ni orden, pero es más rápido, usado en videollamadas y streaming.

623. Cuerpo de la Solicitud (Request Body) en HTTP

890. Deadlock (interbloqueo)

891. Micro Datos

Son otra forma de marcado semántico HTML que permite añadir información estructurada dentro del contenido de una página web para que buscadores y aplicaciones puedan interpretarla mejor.

892. Starvation (inanición)

Cuando un proceso nunca obtiene recursos necesarios porque otros lo acaparan; se mitiga con políticas de prioridad justas.

893. Favicons

Ses el icono pequeño que aparece junto al título de una página web en la pestaña del navegador, y también puede aparecer en marcadores, historial o accesos directos.

894. Thread pool (piscina de hilos)

Grupo de hilos reutilizables para ejecutar tareas concurrentes evitando costes de creación continua.

895. Reactive programming (programación reactiva)

Paradigma que difunde cambios y eventos por flujos, facilitando aplicaciones asíncronas y con alto nivel de concurrencia.

896. iFrames

Es un elemento HTML que permite insertar otro documento HTML dentro de una página web. En otras palabras, es como una “ventana” que muestra otra página dentro de tu página.

897. Cadena de Suministro de Software (Software Supply Chain)

El conjunto de todas las dependencias (librerías, APIs, módulos, contenedores) de terceros que se usan para construir un sistema de Software y que representan un riesgo clave de Ciberseguridad.

898. Backoff (estrategia de retroceso)

Incrementar el tiempo entre reintentos tras fallos (exponencial) para evitar sobrecargar servicios en recuperación.

899. Patrón de Saga (Microservicios Distribuidos)

Un patrón de diseño de Arquitectura que gestiona la consistencia de Datos en un sistema de Microservicios distribuidos a través de una secuencia de transacciones locales, revirtiendo las anteriores (compensating transactions) si falla una de ellas.

900. Throttling (limitación)

Reducir ritmo de procesamiento o peticiones aceptadas para controlar consumo y mantener estabilidad.

901. Patrón Sidecar (Arquitectura de Microservicios)

Contenedor auxiliar que corre junto a la aplicación principal (en la misma unidad de despliegue) para manejar tareas transversales (ej. logging o seguridad), separando la lógica de negocio.

902. Bulkhead pattern (mamparo)

Aislar componentes del sistema en compartimentos para que fallos en uno no propaguen a otros, como compartimentos estancos.

903. Patrón de Desconexión (Circuit Breaker)

Patrón de Arquitectura de Software que detecta fallos prolongados en un servicio remoto (ej. un Microservicio) y detiene las llamadas a ese servicio para evitar la sobrecarga y permitirle recuperarse.

904. Patrón Higuera Estranguladora (Strangler Fig Pattern)

Estrategia de Arquitectura para modernizar un sistema legado reemplazando sus funcionalidades gradualmente con Microservicios nuevos y Cohesivos, "estrangulando" (aislando) el sistema antiguo hasta su eliminación.

905. Decorator pattern (patrón decorador)

Añade responsabilidades a objetos de forma dinámica envolviéndolos en objetos decoradores sin cambiar su estructura original.

906. Principios SOLID (917-922)

Son una colección de cinco principios individuales, todos nuevos en este contexto de diseño de software.

907. FinOps y Cloud Governance

Conceptos como Usage quotas, Billing alerts, Spot instances y Reserved instances son muy específicos de la gestión de costos en la nube, lo que los diferencia del concepto general de "Alertas" que aparece anteriormente.

908. Capa de Abstracción de Hardware (HAL)

Conjunto de rutinas de **Software** que proporcionan una interfaz uniforme a los componentes de **Hardware**, permitiendo que el **Sistema Operativo** o el software superior sean portables entre diferentes plataformas **CPU/IoT**.

909. Arquitectura Microservices

Un estilo moderno de Software donde una aplicación compleja se construye como una colección de servicios pequeños e independientes, cada uno ejecutándose en su propio proceso y comunicándose a través de APIs. Es el opuesto de la arquitectura Monolítica.

910. Prioridad de Afinidades (Affinity Scheduling)

Técnica de programación del **Sistema Operativo** que intenta mantener un hilo de ejecución en el mismo núcleo de la **CPU** donde se ejecutó previamente para maximizar el uso de la **Caché** local y reducir la **Latencia**.

911. Concepto de Idempotencia (Non-Idempotent) de Clicks

En Lógica de Programación de Web, se refiere a una acción del usuario (como hacer clic en un botón de pago) que, si se repite, causará efectos secundarios indeseados (ej., cobrar dos veces). El Software debe evitar esto mediante mecanismos de control.

912. Sistema Lock-Free (Sin Bloqueos)

Código o estructura de **Datos** concurrente que garantiza que, si se detiene un hilo, el sistema aún puede avanzar, eliminando la posibilidad de **Interbloqueos** y maximizando la **Disponibilidad**.

913.Memoria Dirty (Sucia)

Un bloque de Datos en la RAM o Caché que ha sido modificado por el CPU y es diferente de la copia original almacenada en el HDD/SSD. El Software del Sistema debe asegurarse de que esta copia "sucia" sea escrita antes de ser descartada.

914. Principio de Hard-Real-Time (Tiempo Real Duro)

Un requisito de Viabilidad en el diseño de Software y Hardware (típico en Arduino o sistemas médicos) que dicta que la falla en completar una tarea dentro de un plazo de tiempo estricto se considera un fallo total del sistema.

915. Aserción (Assertion)

Una línea de Lógica de Programación que verifica una condición que el programador espera que siempre sea Verdadera en ese punto del código (ej., que una Variable nunca es Null). Si la aserción falla, el Software se detiene, ayudando al debugging temprano.

916. El Call Stack (Pila de Llamadas)

Una estructura de Datos crucial de la RAM que rastrea la secuencia de Funciones que se han llamado y que aún no han terminado de ejecutarse. Es fundamental para el debugging y para saber dónde debe volver el Software tras completar una función.

917. Time-of-Check to Time-of-Use (TOCTOU)

Una clase de bug de seguridad y Lógica de Programación que ocurre cuando el estado de un recurso cambia entre el momento en que el Software lo comprueba (Verifica) y el momento en que lo utiliza (Usa), lo que puede llevar a una vulnerabilidad.

918. Memoria Resident Set Size (RSS)

Una métrica de Mantenimiento de Linux que mide la cantidad de RAM física (no de swap) que está siendo utilizada activamente por un proceso de Software. Es un indicador clave del consumo de recursos de una aplicación.

919. Tiempo de Setup y Hold

Dos requisitos de timing críticos en el Hardware (circuitos digitales y CPU). El tiempo de Setup es el tiempo mínimo que la señal de Datos debe ser estable antes del pulso de reloj; el tiempo de Hold es el tiempo que debe ser estable después. Fallar en estos Binario causa inestabilidad.

920. Entropía de la Información (Information Entropy)

Medida de la incertidumbre o la cantidad de información en un conjunto de **Datos**. Es clave en la generación de números aleatorios seguros y en la fortaleza de las funciones **Hash** en la **Criptografía**.

921. Memoria Swap o Paginación

Un área del HDD/SSD que el Software del Sistema (Linux) usa como extensión de la RAM cuando esta se agota. Mover Datos entre RAM y Swap (paginación) es muy lento y causa Latencia.

922. Análisis Estático de Seguridad de Aplicaciones (SAST)

Metodología de **Testing** que analiza el **código fuente** del **Software** (o binarios) **sin ejecutarlo**, buscando vulnerabilidades conocidas y fallos de **Ciberseguridad** antes del despliegue.

923. Codificación URL Canonicalization (Canonización)

El proceso de normalizar o convertir una URL a su forma más simple y preferida (ej., asegurando que siempre use https y que no tenga barras duplicadas). Es clave para el Mantenimiento de la Web y el SEO.

924. El Critical Path (Ruta Crítica)

Una técnica de Gestión de Proyectos donde se identifica la secuencia más larga de tareas dependientes que deben completarse para que el proyecto finalice. Es la ruta que tiene cero holgura y determina la Deadline mínima.

925. Polimorfismo Ad Hoc (Ad Hoc Polymorphism)

Tipo de **Polimorfismo** donde una misma función o **Operador** puede actuar sobre distintos tipos de **Datos**, pero su implementación específica es decidida durante el tiempo de compilación (**Sobrecarga de Funciones**).

926. Principio de YAGNI (You Aren't Gonna Need It)

Una filosofía de Lógica de Programación y Gestión de Proyectos que aconseja no añadir funcionalidad hasta que sea absolutamente necesario. Esto reduce el Alcance innecesario y la Deuda Técnica.

927. Principio de Referential Transparency

Una propiedad de la Lógica de Programación que significa que una expresión puede ser reemplazada por su valor sin afectar el comportamiento del Software en absoluto. Esto facilita la Optimización y las pruebas (es la meta de las Funciones sin Efectos Secundarios).

928. Patrón Fluent Interface (Interfaz Fluida)

Un estilo de Lógica de Programación donde los métodos de un objeto están diseñados para devolver siempre el propio objeto, permitiendo encadenar múltiples llamadas a métodos en una única línea de Software (ej.: objeto.hacer().conectar().validar();), mejorando la legibilidad.

929. Ataque de Clickjacking (Secuestro de Clics)

Una vulnerabilidad de Web donde un atacante utiliza un iframe transparente y malicioso sobre un botón legítimo de tu Landing Page. La víctima cree que está haciendo clic en tu sitio, pero en realidad está autorizando una acción maliciosa.

930. Ataque de Cross-Site Request Forgery (CSRF)

Una vulnerabilidad de Web donde un atacante engaña al navegador de un usuario autenticado para que envíe una solicitud HTTP maliciosa a un Endpoint (ej., cambiar su contraseña), sin que el usuario lo sepa.

931. Principio Least Recently Used (LRU)

Un algoritmo de Software del Sistema para gestionar la Caché que, cuando la memoria está llena, elimina el bloque de Datos que fue accedido hace más tiempo. Se basa en la suposición de que los datos más antiguos son los menos probables de ser necesarios de nuevo.

932. Principio de Demeter (Law of Demeter - LoD)

Una regla de Lógica de Programación que dicta que un objeto solo debe hablar con sus "amigos inmediatos" y no con los amigos de sus amigos, reduciendo el Acoplamiento y mejorando el Mantenimiento.

933. Efecto Snowflake (Copo de Nieve)

Un término de Mantenimiento para describir sistemas de Hosting o servidores que fueron configurados manualmente y nunca replicados, lo que hace que cada servidor sea único (como un copo de nieve) y extremadamente difícil de auditar o replicar en caso de fallo.

934. Lenguaje Ubicuo (Ubiquitous Language - DDD)

Términos comunes y consensuados utilizados por los expertos del dominio y los desarrolladores para describir el modelo de negocio, eliminando ambigüedades en la comunicación y el código.

935. Diseño Evolutivo (Evolutionary Design)

Enfoque de **Arquitectura de Software** que permite que el diseño cambie y se adapte de forma incremental a lo largo del tiempo, basado en la retroalimentación continua y los requisitos cambiantes. Es la base de las **Metodologías Ágiles**.

936. Sistema de Requisitos No Funcionales (NFRs)

Especificaciones que definen criterios operativos (ej. **Rendimiento**, **Escalabilidad**, Usabilidad, **Seguridad**) más que una función específica; son vitales para la calidad y la arquitectura del sistema.

937. Análisis de Impacto (Impact Analysis)

Proceso formal para evaluar las posibles consecuencias de un cambio propuesto (ej. requisito, **Bug fix**, actualización tecnológica) en los componentes y procesos existentes del sistema de **Software**.

938. Forking en GitHub (Software)

Tabla que vincula cada requisito de **Software** con las pruebas, el código o los casos de uso correspondientes, garantizando que todos los requisitos sean probados y cumplidos a lo largo del ciclo de vida.

939. Proceso de Triage de Bugs (Bug Triage)

Proceso de clasificación y priorización de **Bugs** (por gravedad, frecuencia, impacto) por parte del equipo de desarrollo, decidiendo si se solucionan inmediatamente, se posponen o se cierran.

940. Compilación Ahead-Of-Time (AOT)

Un método de Software del Sistema donde el código se compila directamente a Código de Máquina antes de que se ejecute, a diferencia de JIT. Esto elimina el tiempo de compilación en tiempo de ejecución, mejorando la Latencia inicial.

941. Pruebas de Mutación (Mutation Testing)

Tipo de **Testing** que evalúa la calidad de las pruebas unitarias al introducir pequeños errores (*mutaciones*) en el código y verificar si las pruebas existentes son capaces de detectarlos y fallar.

942. Etiqueta de Entidad (ETag)

Un identificador único que se adjunta al Encabezado HTTP de un recurso Web por el servidor. Permite a los navegadores (clientes) solicitar el recurso y validar rápidamente si su versión en Caché sigue siendo válida sin tener que descargar el contenido completo, mejorando el Rendimiento.

943. Pruebas de Carga (Load Testing)

Tipo de **Testing de Rendimiento** para verificar el comportamiento de un sistema bajo la carga de usuarios esperada en condiciones normales, asegurando que cumple con los **SLAs**.

944. Registro de Instrucciones (Instruction Register - IR)

Definición: Un **registro de Hardware crucial dentro del CPU que contiene la instrucción Binaria que se está ejecutando en ese momento. El CPU lee una instrucción de la RAM, la almacena aquí y luego la decodifica y ejecuta, siendo fundamental para el ciclo de vida del Software.

945. Compilador de Bytecode

Un tipo de código intermedio de Software del Sistema que se genera después de la compilación, pero antes de convertirse en Código de Máquina nativo. Este bytecode está optimizado para ejecutarse en una Máquina Virtual (VM), como la JVM de Java, permitiendo la portabilidad entre diferentes CPU.

946. Fuzz Testing (Pruebas de Fuzzing)

Técnica de **Seguridad** que inyecta **Datos** aleatorios o semi-válidos (*fuzz*) a las entradas de un programa para encontrar errores de desbordamiento, **Excepciones** o fallos de **Seguridad**.

947. Patrón de Desconexión (Circuit Breaker)

Mecanismo que detecta fallos prolongados en un servicio remoto y "abre" el circuito, deteniendo las llamadas subsecuentes para evitar la sobrecarga y permitir la recuperación del servicio.

948. Aislamiento de Fallos (Fault Isolation)

Diseño de **Arquitectura** que contiene un fallo dentro de un componente o subsistema específico (ej. usando el patrón **Bulkhead**), impidiendo que se propague y cause una interrupción total del servicio.

949. Bus SPI (Serial Peripheral Interface)

Un protocolo de comunicación serial síncrono de Hardware (común en Arduino) que utiliza cuatro cables (**Reloj, Datos de entrada, Datos de salida y selección de chip). Es notablemente más rápido que el protocolo I²C para la transferencia de Datos entre CPU y periféricos.

950. Concepto de Kernel Panic (Pánico del Núcleo)

Una acción de seguridad del Sistema Operativo (Linux) que ocurre cuando el Software del Sistema detecta un error fatal del cual no puede recuperarse (ej., corrupción crítica de la RAM o fallo de un subsistema del CPU). El sistema se detiene completamente para prevenir más daños.

951. Pool de Hilos (Thread Pool)

Un patrón de Software del Sistema para la concurrencia que mantiene un conjunto precreado de hilos de ejecución (threads) listos para recibir y ejecutar tareas. Esto reduce la sobrecarga de crear y destruir threads constantemente, mejorando el Rendimiento del Hosting en servidores ocupados.

952. Cohesión Funcional (Functional Cohesion)

El nivel más alto y deseable de **Cohesión** en un módulo de **Software**, donde todos los elementos de ese módulo contribuyen a realizar una única tarea bien definida y con un propósito singular.

953. Estructura de Datos Heap (Montículo)

Una estructura de Datos de Lógica de Programación que satisface la propiedad de heap (cada nodo padre es mayor o igual que sus hijos). Se utiliza para implementar colas de prioridad y es la base de los algoritmos de Ordenamiento (como Heapsort).

954. Acoplamiento de Control (Control Coupling)

Ocurre cuando un módulo le pasa a otro una bandera o un valor que dicta la **lógica de ejecución** interna del módulo receptor, lo que aumenta el acoplamiento y reduce la flexibilidad.

955. Propagación de Excepciones (Exception Propagation)

Mecanismo por el cual un error no manejado asciende en la **Pila de Llamadas** (call stack) hasta que es capturado por un bloque `catch` o llega al nivel superior y detiene la aplicación.

956. Programación Basada en Eventos (Event-Driven Programming)

Paradigma de **Programación** donde el flujo del programa es determinado por eventos (ej. clic de ratón, llegada de mensaje, lectura de sensor) y el código reacciona a estos eventos de forma asíncrona.

957. Balanceo de Carga Round Robin

Algoritmo de **Balanceo de Carga** simple que distribuye el tráfico a los servidores de forma secuencial, uno por uno, en un ciclo rotatorio, sin considerar la carga o capacidad de cada servidor.

958. Principio de Fail-Fast (Falla Rápida)

Una filosofía de Lógica de Programación que exige que el Software falle inmediatamente y visiblemente en cuanto detecta un error, en lugar de intentar recuperarse o continuar con un estado corrupto. Esto facilita el debugging y la Viabilidad a largo plazo.

959. Balanceo de Carga por Mínimas Conexiones

Algoritmo de **Balanceo de Carga** más inteligente que el **Round Robin** que dirige el nuevo tráfico al servidor con el **menor número de conexiones activas** en ese momento.

960. Memoria Slab Allocation (Asignación por Losa)

Una técnica avanzada de Software del Sistema en Linux para gestionar la RAM de objetos de tamaño similar, preasignando losas de memoria. Esto reduce la fragmentación de memoria y mejora la velocidad de asignación y liberación de recursos.

961. Protocolo XMPP (Extensible Messaging and Presence Protocol)

Un protocolo de Protocolo IP basado en XML diseñado para la mensajería instantánea y la información de presencia. Es descentralizado y extensible, usado en sistemas de chat y federación de Datos, aunque es más pesado que protocolos ligeros como MQTT.

962. Política de Retención de Datos (Data Retention)

Reglas de negocio y legales que definen por cuánto tiempo y bajo qué condiciones deben almacenarse ciertos tipos de **Datos** (legales, financieros, personales) antes de su destrucción o archivado.

963. Problema del Lector-Escritor (Readers-Writers)

Problema clásico de **Sincronización** de **Procesos** donde múltiples hilos pueden leer datos concurrentemente, pero la escritura debe ser exclusiva para evitar la corrupción de los **Datos** y las **Condiciones de Carrera**.

964. Idempotencia (Idempotence)

Propiedad de una operación o función donde la ejecución repetida con los mismos parámetros produce el mismo resultado y estado del sistema. Es clave en **APIs** y sistemas de colas de mensajes.

965. El Linker (Enlazador)

Un componente del Software del Sistema (y del proceso de Compilación) que toma uno o más archivos de código objeto (ya compilados) y bibliotecas, y los combina en un único programa ejecutable final (Binario), resolviendo las referencias a Funciones externas.

966. Estructura Inode (Nodo Índice)

Una estructura de Datos fundamental de los Sistemas Operativos (Linux y Unix) que almacena toda la información clave sobre un archivo o directorio (metadatos, permisos, tamaño, propietario) excepto el nombre del archivo y el contenido real. .

967. Patrón Decorator (Decorador)

Un Patrón de Diseño de Software estructural que permite añadir dinámicamente nuevas responsabilidades a un objeto existente en tiempo de ejecución, envolviendo el objeto original con una clase wrapper (decoradora) que ofrece la funcionalidad extendida.

968. Estrategia de Ramificación Trunk-Based Development

Práctica de **Control de Versiones** donde los desarrolladores fusionan los cambios directamente en la rama principal (*trunk*) al menos una vez al día, usando **Feature Flags** para ocultar funciones incompletas.

969. Estrategia Feature Flag (Interruptor de Funcionalidad)

Técnica de **DevOps** que permite activar o desactivar una nueva funcionalidad del **Software** en producción mediante un interruptor de configuración, desacoplando el despliegue de la liberación de la función.

970. Estructura de Datos Hash Map / Diccionario

Una estructura de Datos fundamental que asocia claves únicas con valores para permitir la recuperación de Datos de forma extremadamente rápida. Utiliza una función hash para calcular la posición de memoria de un elemento directamente, ofreciendo un Rendimiento promedio de O(1).

971. Metodología SAFe (Scaled Agile Framework)

*Framework* ágil que se utiliza para aplicar prácticas *lean* y ágiles a **grandes organizaciones** que requieren coordinar cientos o miles de personas en múltiples equipos y **Trenes de Liberación**.

972. Metodología Crystal (Metodología Ágil)

Familia de metodologías ágiles ligeras enfocadas en la interacción humana, la comunicación y la entrega frecuente, clasificadas por el "color" de la criticidad y tamaño del proyecto.

973. Memoria de Página Cero

Un bloque de RAM que ha sido llenado completamente con ceros binarios por el Software del Sistema antes de ser asignado a un nuevo proceso. Es una medida de seguridad esencial para garantizar que los nuevos programas no puedan leer Datos sensibles dejados por procesos anteriores.

974. Registros de Barrido (Scratchpad Registers)

Un tipo de **registro de Hardware de la CPU que el microcódigo o el Compilador pueden usar como espacio temporal de trabajo y que no son visibles para el Software que se ejecuta directamente. Es el almacenamiento más rápido dentro del chip del CPU para operaciones intermedias.

975. Atracción de Fallos (Fault Magnetism)

Fenómeno observado en sistemas complejos donde una porción de código o un componente que ha tenido fallos previamente es más probable que desarrolle más fallos en el futuro, afectando el **Mantenimiento**.

976. Refactorización

Una práctica de Mantenimiento de Software donde se reestructura y limpia el código interno sin cambiar su funcionalidad externa (comportamiento). El objetivo es mejorar la calidad, la legibilidad y reducir la Deuda Técnica, lo que permite añadir nuevas Funciones más fácilmente después.

977. Protocolo OSPF (Open Shortest Path First)

Un protocolo de enrutamiento interno utilizado por routers para determinar el camino más eficiente dentro de un gran Sistema Autónomo (AS). Funciona creando un mapa del grafo de la red y utilizando un algoritmo para calcular la mejor ruta de Datos en tiempo real.

978. Asignación de Registros (Register Allocation)

Una fase de Optimización del Compilador donde se decide **qué Variables de Lógica de Programación deben almacenarse en los Registros de Hardware de la CPU (la memoria más rápida) y cuáles deben almacenarse en la RAM (Stack). Es una de las optimizaciones más críticas para el Rendimiento final.

979. Protocolo QUIC (Quick UDP Internet Connections)

Un protocolo de transporte desarrollado por Google que corre sobre UDP y que está diseñado para reducir la Latencia de conexión HTTP. Combina los beneficios de TCP (confiabilidad, control de congestión) con multiplexación y cifrado integrados desde el primer handshake.

980. Principio Domain-Driven Design (DDD)

Una filosofía de Arquitectura de Software que vincula la implementación del Software con un modelo en constante evolución del dominio central del negocio. Se centra en el uso de un Lenguaje Ubicuo compartido entre los expertos en el negocio y los desarrolladores.

981. Rolling Wave Planning (Planificación de Onda Continua)

Técnica de **Gestión de Proyectos** donde el trabajo a corto plazo se planifica con gran detalle, mientras que el trabajo a largo plazo se planifica de forma más general, adaptándose a la incertidumbre.

982. Gestión de Riesgos con Sprint Zero

Un *Sprint* inicial no productivo, de duración fija, dedicado a la preparación de la infraestructura (**DevOps**) o la mitigación de riesgos técnicos importantes antes de iniciar el desarrollo del producto.

983. Arquitectura Harvard

Un diseño de Arquitectura de Hardware (común en Arduino y procesadores de señales digitales) que utiliza dos espacios de memoria separados (y buses separados) para almacenar Instrucciones y Datos. Esto permite que el CPU acceda a Datos e Instrucciones simultáneamente, mejorando drásticamente el Rendimiento y el Pipelining.

984. Concepto de Currying (Currificación)

Una técnica de Lógica de Programación funcional donde una Función que toma múltiples argumentos es transformada en una secuencia de Funciones donde cada una solo toma un argumento. Esto facilita la reutilización y la creación de Funciones de orden superior.

985. Codificación de Golomb (Golomb Coding)

Método de **Compresión de Datos** sin pérdida utilizado para codificar números enteros positivos, siendo especialmente eficiente cuando los valores pequeños son más probables que los grandes.

986. Polinomios de Redundancia Cíclica (CRC)

Código de detección de errores utilizado en la transmisión de **Datos** y **Almacenamiento** para detectar cambios accidentales. Se basa en la división polinomial de los datos de entrada.

987. Aliasing de Punteros (Pointer Aliasing)

Un problema de Lógica de Programación donde dos o más Punteros apuntan a la misma ubicación de RAM. Esto complica la Optimización del Compilador, ya que cualquier cambio a través de un Puntero podría afectar al valor referenciado por el otro.

988. Capa de Proxy Inverso (Reverse Proxy)

Servidor que se coloca frente a los servidores web para recibir peticiones de clientes, proporcionando servicios de **Seguridad**, **Balanceo de Carga** y **Caching** antes de reenviar la petición al servidor.

989. Lista Enlazada Doblemente Circular

Una Estructura de Datos donde cada nodo tiene Punteros tanto al nodo siguiente como al nodo anterior, y el nodo final apunta de vuelta al primer nodo. Esto permite recorrer la lista en ambas direcciones y en un Bucle Infinito con facilidad.

990. Patrones de Diseño

Un Patrón de Diseño de Software de comportamiento que permite definir una familia de algoritmos, encapsular cada uno en una clase separada, y hacer que sean intercambiables por el cliente en tiempo de ejecución. Esto permite al Software elegir el mejor algoritmo de Ordenamiento sin modificar su código.

991. Patrón de Sello de Tiempo (Timestamping)

Adición de un valor que representa el momento exacto en que un evento ocurrió o un **Dato** fue creado/modificado. Es esencial para la **Trazabilidad** y la resolución de conflictos en **Bases de Datos** distribuidas.

992. Arquitectura de Cuatro Capas (4-Tier Architecture)

Un patrón de diseño que extiende la arquitectura de tres capas, añadiendo una capa de Middleware o servicios entre la capa de Presentación (Web) y la capa de Lógica de Negocio. Se utiliza en sistemas muy grandes para manejar la complejidad de la lógica de negocio distribuida.

993. Métrica Mean Time Between Failures (MTBF)

Una métrica clave de Mantenimiento que mide el tiempo promedio que se espera que un sistema, Hardware o componente opere sin fallar. Se utiliza para predecir la Viabilidad del equipo y planificar el Mantenimiento preventivo..

994. El Modelo Entidad-Relación (MER)

Es una herramienta fundamental y conceptual utilizada en el diseño de bases de datos para representar la estructura lógica de los datos y sus interconexiones en un sistema.

995. Entidad (Entity)

Una Entidad representa un objeto o concepto del mundo real sobre el cual deseamos almacenar información. Estos pueden ser concretos (una persona, un producto) o abstractos (un evento, un cargo). En el modelo final, cada entidad generalmente se traduce en una tabla en la base de datos.

996. Atributo (Attribute)

Un Atributo es una propiedad o característica descriptiva de una entidad. Cada instancia de la entidad tendrá un valor específico para cada uno de sus atributos. Los atributos se traducen en las columnas de una tabla.

997. Relación (Relationship)

Una Relación es la asociación o vínculo entre dos o más entidades. Define cómo las instancias de una entidad interactúan con las instancias de otra.

998. Clave Candidata (Candidate Key)

Una Clave Candidata es cualquier atributo (o conjunto mínimo de atributos) dentro de una Entidad que puede utilizarse para identificar de forma única cada una de sus instancias.

999. Clave Primaria (Primary Key - PK)

La Clave Primaria es la Clave Candidata que el diseñador de la base de datos elige para ser el identificador principal y oficial de una entidad.

1000. Super Clave (Super Key)

Una Super Clave es cualquier atributo (o conjunto de atributos) que, por sí solo o en combinación, permite identificar de forma única cada tupla (fila) en una relación (tabla).

Claves

Las Claves son los atributos o conjuntos de atributos que nos permiten identificar a cada fila de forma única dentro de una entidad, asegurando que no haya confusión.

Clave Primaria (PK): Es el identificador único y OFICIAL que elegimos para la entidad. Es como el DNI de esa tabla. En el diagrama, se subraya. Ejemplo en Proyecto: El atributo ID se elegiría como Clave Primaria.
Clave Candidata: Son todos los atributos que podrían ser la Clave Primaria porque son únicos y mínimos (no tienen información extra). Ejemplo en Desarrollador/a: Si tuviéramos DNI y Email, ambas serían claves candidatas, pero solo elegimos una como Primaria.
1 Super Clave: Es cualquier conjunto de atributos que incluye una Clave Candidata. Es la regla de unicidad más amplia. Puede contener datos innecesarios, pero sigue garantizando que la fila es única.
